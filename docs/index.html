MiniBus architecture primer

Azure-native messaging without handler pollution.

MiniBus is a compact .NET message-processing framework for Azure Functions and Azure Service Bus. It keeps Azure Functions triggers and Azure SDK types at the edge, while business code works with message contracts, handlers, sagas, and MiniBusContext.

.NET / C# Azure Functions isolated worker Azure Service Bus Handlers + sagas SQL inbox/outbox Blob claim-check/audit
Thin adapter Function triggers delegate to MiniBusProcessor; handlers stay transport-neutral.
Explicit core Contracts, routing, persistence and recoverability are configured in application startup.
Reliability path Inbox, outbox and saga state can be persisted to SQL; large payloads can use Blob claim-check.

1. Mental model

Treat MiniBus as a small application-level bus runtime. Azure Functions and Azure Service Bus provide hosting and transport. MiniBus provides the contract model, processing pipeline, handler invocation, saga invocation, routing, recoverability hooks, and optional persistence.

Edge

Azure Function wrapper

Contains the actual [ServiceBusTrigger] declaration and forwards the received message plus settlement actions to MiniBus.

Runtime

MiniBusProcessor

Maps headers, resolves message type, deserializes, invokes handlers and sagas, commits persistence state, dispatches outgoing operations, then settles.

Application

Handlers and sagas

Implement business behavior through IHandleMessages<T> and IHandleSagaMessages<T>. They use MiniBusContext for outgoing messages.

Design rule: application handlers should not depend on ServiceBusReceivedMessage, ServiceBusMessageActions, Azure Functions binding attributes, or Azure SDK transport details.

2. What happens under the hood

ReceiveAzure Functions receives a Service Bus message and calls the wrapper.
AdaptThe adapter maps the Azure message into MiniBus processing inputs.
Resolve + deserializeHeaders identify the message type and payload is deserialized through the configured serializer.
Check inboxWhen SQL persistence is enabled, duplicate messages can be skipped before handlers run.
Invoke handlers and sagasHandlers and sagas can publish, send, schedule and request saga timeouts through MiniBusContext.
Commit + dispatch + settleMiniBus commits persistence state, captures or dispatches outgoing work, then completes, retries, schedules retry, dead-letters, or propagates failure.

3. Project/package map

The solution is divided into small packages. Architecturally, this makes the application decide how much runtime surface it wants: only core contracts, Functions hosting, Service Bus transport, SQL reliability, Azure Storage payload/audit support, tooling, analyzers, templates, or testing helpers.

Contracts, handlers, context, serialization, recoverability, saga abstractions and persistence abstractions.

4. Developing handlers

A handler is deliberately small: it receives a deserialized contract, MiniBusContext, and a cancellation token. The context exposes outgoing message operations, so the handler does not need Service Bus sender clients or Function trigger artifacts.

Handler shape

public sealed record CreateInvoice(
    string InvoiceId,
    string CustomerId,
    decimal Amount) : ICommand;

public sealed class CreateInvoiceHandler
    : IHandleMessages<CreateInvoice>
{
    public async Task Handle(
        CreateInvoice message,
        MiniBusContext context,
        CancellationToken cancellationToken)
    {
        await context.Publish(
            new InvoiceCreated(
                message.InvoiceId,
                message.CustomerId,
                message.Amount),
            cancellationToken);

        await context.Send(
            new ReserveInventory(
                message.InvoiceId,
                message.CustomerId,
                "sample-sku",
                1),
            cancellationToken);
    }
}

Handler rules

Use contractsDefine messages as ICommand, IEvent, or IMessage.
Use DIRegister handlers explicitly, or use the discovery/registration style the application adopts.
Use MiniBusContextCall Send, Publish, and Schedule from context instead of constructing broker messages.
Keep idempotencyOutbox dispatch is at-least-once. Downstream handlers should tolerate duplicates; SQL inbox helps with receive-side deduplication.
Test directlyUnit test handler logic without Azure Functions host, Service Bus emulator, or Azure SDK objects unless infrastructure behavior is under test.

5. Developing sagas

Sagas model long-running workflows. MiniBus saga support is explicit: define saga data, inherit from MiniBusSaga<TData>, implement IHandleSagaMessages<TMessage>, configure correlation rules, and register saga mappings at startup.

Saga shape

public sealed class BillingSagaData : ISagaData
{
    public Guid Id { get; set; }
    public string CorrelationId { get; set; } = "";
    public bool IsCompleted { get; set; }
    public bool InvoiceCreated { get; set; }
    public bool PaymentTimedOut { get; set; }
}

public sealed class BillingSaga :
    MiniBusSaga<BillingSagaData>,
    IHandleSagaMessages<InvoiceCreated>,
    IHandleSagaMessages<InvoicePaymentTimeout>
{
    public override void ConfigureHowToFindSaga(
        SagaMapper<BillingSagaData> mapper)
    {
        mapper.StartsWith<InvoiceCreated>(m => m.InvoiceId)
              .Correlate<InvoicePaymentTimeout>(m => m.InvoiceId);
    }

    public async Task Handle(
        InvoiceCreated message,
        MiniBusContext context,
        CancellationToken cancellationToken)
    {
        Data.InvoiceCreated = true;

        await RequestTimeout(
            new InvoicePaymentTimeout(message.InvoiceId),
            TimeSpan.FromDays(7),
            context,
            cancellationToken);
    }

    public Task Handle(
        InvoicePaymentTimeout message,
        MiniBusContext context,
        CancellationToken cancellationToken)
    {
        Data.PaymentTimedOut = true;
        MarkAsComplete();
        return Task.CompletedTask;
    }
}

Saga lifecycle

Registration

services.AddMiniBusAzureFunctions(options =>
{
    options.EndpointName = "Billing";
    options.EnableSagas = true;
});

var sagaRegistry = new SagaRegistry();
sagaRegistry.Register<BillingSaga, BillingSagaData>();

services.AddSingleton(sagaRegistry);
services.AddSingleton<SagaInvoker>();

// samples/tests:
services.AddSingleton<ISagaPersistence, InMemorySagaPersistence>();

// production reliability path:
services.AddMiniBusSqlPersistence(connectionString);
Production note: in-memory saga persistence is for samples and tests. SQL persistence provides production saga persistence with optimistic concurrency.

7. Reference startup composition

A typical Azure Functions endpoint composes the adapter, serializer, handlers, routes, transport dispatcher, Service Bus client, delayed retry scheduler, saga registry and optional SQL persistence. 

services.AddMiniBusAzureFunctions(options =>
{
    options.EndpointName = "Billing";
    options.EnableSagas = true;
    options.Recoverability.ImmediateRetries = 3;
    options.Recoverability.DelayedRetries.Add(TimeSpan.FromSeconds(10));
    options.Recoverability.DelayedRetries.Add(TimeSpan.FromMinutes(1));
    options.Recoverability.DelayedRetries.Add(TimeSpan.FromMinutes(5));
    options.Recoverability.DeadLetterAfterRetriesExhausted = true;
});

services.AddSingleton<IMessageSerializer, SystemTextJsonMessageSerializer>();
services.AddTransient<IHandleMessages<CreateInvoice>, CreateInvoiceHandler>();

var routes = new AzureServiceBusTransportRoutes();
routes.MapCommand<CreateInvoice>("billing-queue");
routes.MapCommand<ReserveInventory>("inventory-queue");
routes.MapEvent<InvoiceCreated>("domain-events");
routes.MapScheduledMessage<InvoicePaymentTimeout>("billing-timeouts");

services.AddSingleton(routes);
services.AddSingleton<AzureServiceBusMessageFactory>();
services.AddSingleton<AzureServiceBusTransportDispatcher>();
services.AddSingleton(_ => new ServiceBusClient(serviceBusConnectionString));

8. Operational decisions

Outbox dispatch

Handler execution and outbox dispatch are intentionally separate. Dispatch can run from an owned timer, worker, hosted dispatcher or separate app.

Delayed retries

Immediate retries run inside the same processor call. Delayed retries use scheduled Azure Service Bus message copies while preserving MiniBus metadata.

SQL schema

MiniBus ships explicit SQL scripts. Applications should apply them through normal database deployment flow before enabling SQL persistence.

Claim-check

Large messages can be written to Blob Storage and replaced by compact broker messages; receive-side resolution happens before handlers see the contract.

Audit blobs

Audit writing is opt-in and fail-closed. If audit is enabled and writing fails, the message is not silently completed as if audit succeeded.

Tooling

Initial tooling is SQL-first: inspect inbox, outbox, sagas, message details and correlations without dumping sensitive payloads by default.
