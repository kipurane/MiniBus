# MiniBus Azure Functions Starter

This project is the first MiniBus `dotnet new` starter for an Azure Functions isolated-worker endpoint backed by Azure Service Bus.

## Starter Flow

`StarterInputFunction` receives `SubmitOrder` messages from the `starter-commands` queue and delegates them to `MiniBusProcessor`. `SubmitOrderHandler` stays on MiniBus APIs, handles the command, and publishes `OrderSubmitted` to the `starter-events` topic through the explicit Azure Service Bus route in `MiniBusConfiguration`.

Build the project before connecting cloud resources:

```bash
dotnet build
```

## Service Bus Configuration

`local.settings.json` includes a `ServiceBus` placeholder for local development. Replace it with the Azure Functions app setting value you want the Service Bus trigger and `ServiceBusClient` registration to use. Use a namespace-level Service Bus connection string for an Azure namespace:

```json
"ServiceBus": "Endpoint=sb://<namespace>.servicebus.windows.net/;SharedAccessKeyName=<policy-name>;SharedAccessKey=<key>"
```

The Azure Service Bus Emulator is also a local development option when it is running on the same machine as the generated Function App:

```json
"ServiceBus": "Endpoint=sb://localhost;SharedAccessKeyName=RootManageSharedAccessKey;SharedAccessKey=SAS_KEY_VALUE;UseDevelopmentEmulator=true;"
```

Configure the starter queue and topic in the emulator before running the project, just as you would create the entities in Azure.

The starter names live in `StarterTopology`:

- input queue: `starter-commands`
- published event topic: `starter-events`
- Functions connection setting: `ServiceBus`

Your application owns Azure Service Bus credentials, queue and topic creation, subscriptions for published events, route naming, duplicate-detection choices, deployment, and production observability.

## Next Steps

Manual Service Bus trigger wrappers are generated by default because the transport boundary is visible in source. To switch to source-generated wrappers later, add `MiniBus.AzureFunctions.SourceGenerators` and declare the queue or topic trigger metadata in the Function App assembly.

This starter intentionally omits SQL inbox/outbox/saga persistence. Add `MiniBus.Persistence.Sql`, apply the shipped schema scripts, register persistence, and schedule outbox draining when the endpoint needs those reliability guarantees.

Saga workflows also stay outside the first starter slice. Use the MiniBus Function App sample and saga documentation when a workflow needs correlation, timeout messages, and persistence decisions beyond this one-handler path.
