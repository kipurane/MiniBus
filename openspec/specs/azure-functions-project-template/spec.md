# azure-functions-project-template Specification

## Purpose
Defines the v1 `dotnet new` Azure Functions + Azure Service Bus starter project shape, documentation, packaging, and generated-project verification expectations.

## Requirements
### Requirement: MiniBus provides a focused Azure Functions project template
MiniBus SHALL provide one v1 `dotnet new` project template for a MiniBus Azure Functions isolated-worker application that uses Azure Service Bus as its transport.

#### Scenario: Developer discovers the v1 template
- **WHEN** a developer installs or inspects the MiniBus template package
- **THEN** the Azure Functions starter template is available with documented installation and invocation guidance

#### Scenario: Developer requests template options
- **WHEN** a developer reviews the v1 template surface
- **THEN** wrapper style, saga inclusion, SQL persistence, and multi-endpoint topology are not exposed as v1 template options

### Requirement: Generated output is a buildable Functions host
The Azure Functions project template SHALL generate a complete isolated-worker Function App project that can build without live Azure Service Bus resources.

#### Scenario: Default project is generated
- **WHEN** the default template is instantiated
- **THEN** the output contains the host entry point and Azure Functions host files required for an isolated-worker Function App project

#### Scenario: Generated project is built locally
- **WHEN** repository template verification builds the default generated project against locally available MiniBus packages
- **THEN** the build succeeds without provisioning Azure Service Bus infrastructure

### Requirement: Generated output demonstrates the starter MiniBus message path
The Azure Functions project template SHALL generate starter message contracts, one MiniBus handler, MiniBus processing registration, Azure Service Bus transport registration, explicit route placeholders for generated messages, recoverability configuration, and local documentation describing the generated path.

#### Scenario: Developer inspects starter application code
- **WHEN** a developer reads the generated project
- **THEN** it includes starter contracts and a handler that processes an inbound MiniBus message through handler-facing APIs rather than Azure SDK trigger types

#### Scenario: Developer inspects registration code
- **WHEN** a developer reads the generated MiniBus setup
- **THEN** it shows endpoint registration, serializer registration, handler registration, Azure Service Bus route and dispatcher registration, and recoverability settings

#### Scenario: Developer reads generated documentation
- **WHEN** a developer opens the generated project documentation
- **THEN** it explains the starter message flow, the configuration placeholders, and the application-owned steps required to run beyond a local build

### Requirement: Generated output uses manual Service Bus wrappers by default
The Azure Functions project template SHALL generate a thin manual Service Bus trigger wrapper by default and SHALL keep source-generated wrappers optional.

#### Scenario: Default trigger wrapper is inspected
- **WHEN** a developer reads the default generated Service Bus trigger wrapper
- **THEN** it receives Azure Functions Service Bus trigger arguments and delegates processing and settlement to `MiniBusProcessor.ProcessAsync`

#### Scenario: Source generator path is documented
- **WHEN** a developer reads the generated project guidance
- **THEN** it explains that `MiniBus.AzureFunctions.SourceGenerators` can be adopted later instead of requiring generated wrappers in the v1 template output

### Requirement: Generated configuration separates placeholders from infrastructure ownership
The Azure Functions project template SHALL generate Service Bus configuration placeholders while making Azure namespace, credentials, topology, and deployment ownership explicit.

#### Scenario: Local Service Bus setting is generated
- **WHEN** the default project is instantiated
- **THEN** the generated configuration identifies the Service Bus connection setting name used by the trigger wrapper

#### Scenario: Topology remains application-owned
- **WHEN** a developer reads the generated routes and documentation
- **THEN** queue, topic, subscription, credential, provisioning, and production deployment choices are described as application-owned responsibilities

### Requirement: V1 template keeps persistence and sagas outside the starter output
The Azure Functions project template SHALL omit SQL persistence wiring and saga workflow code from the default generated project while documenting them as extension points.

#### Scenario: Default project omits production persistence wiring
- **WHEN** the default template output is inspected
- **THEN** SQL inbox, outbox, saga persistence registration, schema deployment automation, and outbox-drain scheduling are not generated by default

#### Scenario: Default project omits saga workflow code
- **WHEN** the default template output is inspected
- **THEN** saga data, timeout messages, saga registration, and saga handlers are not generated by default

#### Scenario: Developer seeks follow-up guidance
- **WHEN** a developer reads the generated documentation
- **THEN** it points to MiniBus guidance for SQL persistence, saga workflows, and richer samples without claiming those concerns are enabled by the starter

### Requirement: Generated output uses developer tooling defaults intentionally
The Azure Functions project template SHALL include MiniBus analyzers by default for compile-time guidance and SHALL not require the source-generator package for the default starter path.

#### Scenario: Generated project dependencies are inspected
- **WHEN** a developer reads the generated project file
- **THEN** it references MiniBus runtime packages and `MiniBus.Analyzers` without requiring `MiniBus.AzureFunctions.SourceGenerators`

#### Scenario: Generated project build runs with analyzers
- **WHEN** repository verification builds the default generated project
- **THEN** analyzer inclusion does not prevent the starter output from building

### Requirement: Template behavior is verified through packaging and generation coverage
MiniBus SHALL verify that the Azure Functions project template can be packaged or installed for `dotnet new`, instantiated with its default output, and built as a generated project.

#### Scenario: Template package is verified
- **WHEN** template verification runs
- **THEN** it exercises the template packaging or installability path needed for `dotnet new`

#### Scenario: Template output is verified
- **WHEN** template verification instantiates the default template output
- **THEN** it checks generated-project buildability and the presence of the core starter files or configuration expected by the template contract
