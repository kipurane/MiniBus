## Context

MiniBus currently has SQL persistence for reliability state and Azure Service Bus scheduling for delayed delivery, but Azure Storage support is still only described in the project backlog and architecture notes. The first Azure Storage slice should be narrow: create the package and prove Blob Storage can hold opaque message payload bytes for future claim-check/DataBus behavior.

The storage layer must keep application handlers storage-independent. Handlers should continue to work with MiniBus message contracts and `MiniBusContext`; Azure SDK types should be isolated to the Azure Storage package and registration APIs.

## Goals / Non-Goals

**Goals:**

- Add a new Azure Storage persistence package that can be referenced independently from SQL persistence.
- Provide configuration and dependency injection primitives for Blob-backed payload storage.
- Define a small payload store contract for writing, reading, and deleting payload bytes.
- Use Azure Blob Storage naming and metadata conventions that future claim-check processing can reuse.
- Support Testcontainers-backed Azurite integration tests, with an optional live Azure Storage connection-string fallback, without requiring shared Azure Storage infrastructure for the normal test run.

**Non-Goals:**

- Do not add automatic claim-check behavior to outgoing or incoming message processing.
- Do not alter `MiniBusContext.Send`, `Publish`, or `Schedule`.
- Do not add Table Storage inbox persistence.
- Do not add Table Storage saga persistence.
- Do not add audit blob writing.
- Do not introduce Azure SDK types into message contracts, handlers, saga data, or `MiniBus.Core`.

## Decisions

### Add a dedicated Azure Storage persistence package

Create `MiniBus.Persistence.AzureStorage` with references to `MiniBus.Core`, `Azure.Storage.Blobs`, and dependency injection abstractions. Keep Blob-specific implementation details out of `MiniBus.Core`, `MiniBus.AzureFunctions`, and `MiniBus.Persistence.Sql`.

Alternative considered: put payload storage contracts in `MiniBus.Core`. That would make the abstraction universally available, but it would introduce an API before any non-Azure implementation exists. The first slice can keep the contract in the Azure Storage package and move it later only if another storage provider needs it.

### Start with a Blob payload store contract

Expose a small MiniBus-owned contract, such as `IMiniBusPayloadStore`, with operations to save a payload, open/read a payload by reference, and delete a payload. The stored reference should be MiniBus-owned data such as container name, blob name, content type, length, and expiry metadata, not an Azure SDK object.

Alternative considered: expose `BlobClient` or blob URIs directly. That is simpler, but it leaks storage mechanics into future claim-check processing and makes tests and alternative implementations harder.

### Let the store generate stable, collision-resistant blob names

Blob names should be generated by the store when saving payloads unless a caller explicitly provides a payload id for deterministic tests. A practical shape is a prefix plus date partition plus GUID-style id, for example `payloads/yyyy/MM/dd/{payloadId}.bin`. The design should avoid message-type-specific names because serialized payloads may later come from commands, events, scheduled messages, or stored claim-check bodies.

Alternative considered: use message ids as blob names. That creates useful traceability, but message ids are not always available at the storage abstraction boundary and can collide across endpoints unless every caller supplies enough context.

### Store enough metadata for future claim-check behavior

Blob writes should set content type when provided and store MiniBus metadata such as payload id, created UTC timestamp, optional expires UTC timestamp, and payload length. The first implementation does not need a cleanup worker, but it should preserve expiry metadata so a future cleanup process can delete expired payloads without changing stored blob format.

Alternative considered: keep expiry entirely in configuration and not metadata. That avoids metadata writes, but makes later cleanup and inspection weaker.

### Use Testcontainers-backed Azurite for integration tests

Unit tests should cover option validation, reference shape, and behavior that can run without Azure Storage. Integration tests should use the `Testcontainers.Azurite` module to start an isolated Azurite container and verify real Blob Storage SDK calls against disposable infrastructure. A documented live Azure Storage connection string may be supported as an optional fallback for provider verification. If Docker/Testcontainers is unavailable and no live connection string is configured, integration tests should skip with a clear reason.

Alternative considered: require developers to run Azurite manually before integration tests. That keeps test dependencies explicit, but it is easier to misconfigure and less consistent than the existing SQL Testcontainers testing style.

## Risks / Trade-offs

- Blob payload storage is useful but not yet wired into message processing -> Keep names and references explicitly designed for future claim-check work and document this as a foundation slice.
- Payload references may become public API too early -> Keep the reference small, storage-neutral where possible, and avoid Azure SDK types.
- Expiry metadata without cleanup can create a false sense of lifecycle management -> Document that deletion is explicit in this change and automated cleanup is future work.
- Azurite/live storage differences can hide production issues -> Cover core blob write/read/delete behavior with Testcontainers-backed Azurite and allow an optional live connection string path for provider verification.

## Migration Plan

This change is additive. Existing applications do not need to reference the new package and no existing storage schemas or message formats change. Rollback is removing the package reference and registration from an application; stored blobs are application-owned data and can be deleted according to the application’s retention policy.

## Open Questions

- Should the payload store contract stay in `MiniBus.Persistence.AzureStorage`, or move to `MiniBus.Core` when full claim-check processing is introduced?
- Should the first implementation expose streaming APIs only, byte-array/BinaryData convenience APIs only, or both?
- What default payload container name should MiniBus recommend: `minibus-payloads` or an endpoint-specific name?
